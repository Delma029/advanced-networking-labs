Link parameters: 1 Gbps, ~200ms RTT (100ms delay each direction via netem+tbf)
Calculated BDP: 25,000,000 bytes (~25 MB)

Default Linux buffer limits (this VM):
  net.core.rmem_max: 212,992 bytes (~208 KB)
  net.core.wmem_max: 212,992 bytes (~208 KB)
  tcp_rmem max: 33,554,432 bytes (~32 MB) -- exceeds BDP, should not bottleneck
  tcp_wmem max: 4,194,304 bytes (~4 MB) -- ~6x smaller than BDP, expected bottleneck

Prediction: default single-stream throughput will fall well short of
1 Gbps, limited specifically by the send-side (wmem) buffer ceiling,
not the receive side. Expect throughput roughly proportional to
wmem_max / BDP ratio -- a rough estimate would put single-stream
throughput in the range of 150-200 Mbps rather than the full 1 Gbps,
though the actual congestion-control dynamics (slow start, AIMD) will
determine the real figure.

ACTUAL RESULT: 22.4 Mbit/s average over 20s (vs 1 Gbps available) --
roughly 2% link utilization. Worse than the rough 150-200 Mbps estimate.

Cwnd grew from ~206KB to ~1.42MB over 20 seconds -- steady linear-ish
growth consistent with AIMD, but far too slow relative to the 25MB BDP
target because each growth step only happens once per ~200ms RTT.
This is the RTT-limited window growth problem directly observed: on a
near-zero-RTT LAN, this same growth pattern would fill the pipe in a
fraction of a second; here it would take many more seconds/minutes to
approach the BDP even without the wmem ceiling capping it first.

Two intermittent zero-throughput seconds and retransmissions late in
the run are consistent with the tbf queue (400ms max latency) dropping
packets under load, triggering TCP's loss-based backoff -- a visible
interaction between the synthetic link's queueing limit and TCP's
congestion response, not test noise.

TUNED BUFFER RESULT: 27.9 Mbit/s (vs 22.4 Mbit/s default) -- only a
~25% improvement, far short of the target 1 Gbps despite raising
tcp_wmem's ceiling 8x (4MB -> 32MB).

Cwnd only grew to 1.74MB by test end (vs 1.42MB in the untuned run) --
nearly unchanged. This reveals the buffer ceiling was NOT the actual
bottleneck in a 20-second test window: the real limit is congestion
control's own growth rate. Standard AIMD-style growth (CUBIC default)
takes roughly one RTT per growth step; at ~200ms RTT, filling a 25MB
BDP window would require far more than 20 seconds regardless of how
much buffer ceiling is available. Buffer tuning removes a hard wall,
but doesn't change the speed of the climb toward it.

This is the practical motivation for BBR: rather than growing the
window via loss-based AIMD, BBR estimates bandwidth and RTT directly
and can ramp toward the correct sending rate far faster, independent
of RTT-gated window growth. Testing BBR against this same link and
buffer configuration is the natural next step.
